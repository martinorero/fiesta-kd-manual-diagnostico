        Hardware vs Data · Caso ABS real · Fiesta KD 2015
        01
        // síntoma inicial · data layer
        El dashboard miente… o eso parecía
        El scanner OBD reportó código C0031 — sensor de velocidad rueda delantera izquierda. Lecturas intermitentes, señal errática. Primera hipótesis obvia: falla de sensor, cableado dañado o módulo ABS. Todo apuntaba a un problema de datos.
        En datos diríamos: "el KPI está rojo, el source system tiene un problema". La métrica está fallando → el origen también debe estar fallando. Lógica razonable. Casi siempre equivocada.
        02
        // capa física · hardware layer
        Ruido metálico al frenar: la pista que el scanner no ve
        Más allá del código, había un síntoma físico claro: ruido metálico al frenar a baja velocidad, exactamente donde el ABS se activa para prevenir bloqueo de rueda. El sistema de frenos generaba vibración suficiente para interferir con la lectura del sensor.
        El equivalente en BI: el KPI roto no es porque el ETL falló, sino porque el proceso de negocio upstream está roto. El problema no es el dato, es la realidad que lo genera.
        03
        // diagnóstico diferencial · la checklist
        Pastillas al límite: la causa raíz estaba en el hardware
        Inspección física del conjunto delantero izquierdo: pastillas al límite del desgaste, indicador metálico rozando el disco. Al frenar, la vibración irregular perturbaba la rueda dentada del sensor ABS, generando pulsos erróneos que el módulo interpretaba como falla de sensor.
        En datos
        KPI de rentabilidad cae → asumís bug en DAX → revisás medidas, filtros, relaciones
        ↔
        En el auto
        Código C0031 → asumís sensor malo → comprás sensor, revisás cableado
        La trampa en ambos casos: ir directo a la capa de datos sin validar la capa física. Pastillas $25.000 vs sensor ABS $80.000+. El diagnóstico diferencial correcto ahorra tiempo y plata.
        04
        // root cause fix · resolución
        Cambio de pastillas + discos → C0031 desaparece
        Cambio de pastillas y discos delanteros. Borrado de DTCs. Código C0031 no regresa. La señal del sensor es limpia porque la fuente del ruido físico fue eliminada. El sensor nunca estuvo roto.
        En datos: corregiste el proceso upstream → el KPI se normaliza solo, sin tocar una línea de DAX. El modelo siempre estuvo bien. El problema era la realidad que lo alimentaba.
        // lección universal
        Antes de culpar al dato, validá la realidad que lo genera.
        Aplica igual cuando el scanner dice "sensor roto" que cuando el dashboard dice "el modelo tiene un bug". La capa de datos refleja la realidad — si la realidad está rota, el dato también va a estarlo. Primero revisá el hardware.
